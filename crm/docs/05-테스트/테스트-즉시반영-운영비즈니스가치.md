# 테스트/검증: 즉시 반영 + 운영 비즈니스 가치

## 1. 핵심 테스트 시나리오

### TC-FN-PERM-IM-001 즉시 반영 권한 변경
- 선행: ADMIN 계정 로그인(Access/Refresh 발급)
- 단계
  1. 대상 사용자 목록 조회 (`GET /api/v1/auth/admin/users`)
  2. 대상 사용자 역할 변경 (`PATCH /api/v1/auth/admin/users/:id/role`)
  3. 대상 사용자는 동일 access token으로 관리자 전용 기능 호출
- 기대: 변경 즉시 적용(403 ↔ 200/201 등 권한 반영)

### TC-FN-DELETE-IM-001 Soft Delete 복구
- 대상: `contacts`, `deals`, `tasks` 각 1건씩
- 단계
  1. 삭제 API 호출
  2. 목록 조회에서 제외 확인
  3. `/{id}/restore` 호출
  4. 상세 조회 가능 확인
- 기대: restored 객체의 `deletedAt`가 null로 복구

### TC-HEALTH-IM-001 운영 지표
- `GET /api/v1/health/ready` 응답 검증(정상시 200)
- DB 점검 실패 주입 시 503과 `db: down` 확인
- `GET /api/v1/health/metrics` 응답 항목 존재(업타임/메모리/체크 상태)

## 2. 수동 점검 포인트
- 관리자 역할 변경 즉시 반영이 실제로 토큰 재발급 없이 동작하는지 로그/응답으로 확인
- `admin` 권한 경로가 `adminOnly`와 소유권 체크를 모두 통과하는지 확인

## 3. 보완 제안
- 장기적으로는 `/api/v1/auth/admin/users/:id/role` 변경 감사 로그에 행동코드(`USER_ROLE_CHANGED`) 분리
- `/health/metrics`에 요청 처리량/DB 쿼리 지연 등을 추가

## 4. 실행 결과 요약 (최종)

### 스크립트 실행
- `npm --prefix apps/api run test` → `12` 파일, `98` 테스트 통과
- `npm --prefix apps/api run contract:audit` → 실패 없이 통과

### 반영 항목
- `audit` 미들웨어 테스트 스킵 처리로 DB 미기동 환경에서 감사 로그 예외 로그 감소
- `admin` 라우트(사용자 목록/권한 변경) 테스트 보강
- `contacts`/`tasks`/`deals` 복구 API(soft delete restore) 테스트 보강
- `health` 운영 지표(`ready`, `live`, `metrics`) 반환값 검증 테스트 추가

### 확인된 환경 이슈
- DB가 오프 상태일 때 헬스 체크는 `ready`가 `503`과 `db: down`을 반환하는 동작을 유지
- 포트/IPC 제약으로 `localhost` 실제 바인딩 기반 실행은 기존 제약 하에서 제한될 수 있음
