# VIVE CRM 우선순위 로드맵 (Priority Roadmap)

> SEO 가능한 Next.js SaaS로의 전환을 위한 실행 로드맵

| 항목 | 내용 |
|------|------|
| **문서 버전** | v1.0 |
| **작성일** | 2026-03-17 |
| **작성자** | 권영해 / 기획·개발 |
| **상태** | 실행 대기 |

---

## 개요

본 로드맵은 현재 VIVE CRM을 SEO 가능한 Next.js SaaS로 전환하기 위한 6단계 우선순위를 수립하고, 각 단계별 구체적인 실행 항목을 정의합니다.

---

## 로드맵 개요

```
Phase 1: 구조 재설계 (2주)
    └── 랜딩/앱 분리, Route Group 설정
            ↓
Phase 2: SEO 구현 (2주)
    └── 메타데이터, OG, Sitemap, robots.txt
            ↓
Phase 3: 보안 강화 (2주)
    └── 인증 모델 개선, CSRF 방어
            ↓
Phase 4: 인프라 개선 (1주)
    └── Redis Rate Limit, 보안 헤더
            ↓
Phase 5: SaaS 모델링 (4주)
    └── Workspace, 멀티테넌트, 구독
            ↓
Phase 6: 운영 보안 (지속)
    └── 감사로그, 모니터링, 알림
```

---

## Phase 1: 구조 재설계 (2주)

### 목표
- `/`를 공개 랜딩으로 전환
- 앱 영역을 `/app/*`으로 분리
- Next.js Route Group 구조 설정

### 작업 항목

| ID | 작업 | 상세 내용 | 담당 | 예상 소요 | 산출물 |
|----|------|-----------|------|-----------|--------|
| P1-1 | Route Group 생성 | `(landing)`, `(app)` 그룹 생성 | FE | 1일 | 디렉토리 구조 |
| P1-2 | 랜딩 페이지 마이그레이션 | 기존 `/` → `/app/dashboard`, 새 `/` 랜딩 생성 | FE | 2일 | 랜딩 페이지 |
| P1-3 | 인증 미들웨어 설정 | `/app/*` 접근 시 인증 체크 | FE | 1일 | middleware.ts |
| P1-4 | 레이아웃 분리 | 랜딩용 레이아웃, 앱용 레이아웃 분리 | FE | 1일 | layout.tsx |
| P1-5 | 네비게이션 정리 | 랜딩 네비 vs 앱 네비 분리 | FE | 1일 | Navigation 컴포넌트 |
| P1-6 | 리다이렉션 설정 | 인증 상태별 리다이렉션 로직 | FE | 1일 | 리다이렉션 규칙 |
| P1-7 | 테스트 | 랜딩/앱 분리 기능 테스트 | QA | 2일 | 테스트 결과 |

### 성공 기준
- [ ] 비회원이 `/` 접속 시 랜딩 페이지 표시
- [ ] 회원이 `/` 접속 시 `/app/dashboard`로 리다이렉션
- [ ] `/app/*` 비회원 접속 시 로그인 페이지로 리다이렉션
- [ ] 랜딩과 앱의 레이아웃이 완전히 분리됨

### 리스크
| 리스크 | 영향 | 대응 방안 |
|--------|------|-----------|
| 기존 사용자 북마크 깨짐 | 중 | /에서 /app/dashboard로 리다이렉션 설정 |
| SEO 권한 분실 | 중 | 기존 /의 콘텐츠를 랜딩에 유지, canonical 설정 |

## Phase 3: 보안 강화 (2주)

### 목표
- 인증 모델 개선 (localStorage → 메모리 + HttpOnly Cookie)
- CSRF 방어 구현
- 세션 보안 강화

### 작업 항목

| ID | 작업 | 상세 내용 | 담당 | 예상 소요 | 산출물 |
|----|------|-----------|------|-----------|--------|
| P3-1 | Auth Store 수정 | localStorage 제거, 메모리 상태로 변경 | FE | 1일 | auth-store.ts |
| P3-2 | API 클라이언트 수정 | withCredentials 설정, 쿠키 자동 전송 | FE | 1일 | api-client.ts |
| P3-3 | 백엔드 쿠키 설정 | Refresh Token HttpOnly + Secure + SameSite | BE | 1일 | 인증 엔드포인트 |
| P3-4 | CSRF Middleware | Sec-Fetch-Site, Origin 검증 | FE/BE | 1일 | middleware.ts |
| P3-5 | CSRF Token | 민감 작업용 CSRF Token 발급/검증 | BE | 1일 | CSRF 토큰 로직 |
| P3-6 | 로그인 잠금 | 5회 실패 시 30분 잠금 구현 | BE | 1일 | 로그인 로직 |
| P3-7 | 세션 만료 | 30분 비활동 만료 구현 | BE | 0.5일 | 세션 미들웨어 |
| P3-8 | 기기 제한 | 최대 3기기 동시 세션 관리 | BE | 1일 | 세션 관리 |
| P3-9 | 이메일 인증 | 가입 후 이메일 인증 필수화 | BE | 1일 | 이메일 인증 |
| P3-10 | 보안 테스트 | XSS, CSRF, 세션 하이재킹 테스트 | QA | 1일 | 보안 테스트 결과 |

### 성공 기준
- [ ] Access Token이 localStorage에 저장되지 않음 (DevTools 확인)
- [ ] Refresh Token이 HttpOnly 쿠키로만 전달됨 (Network 탭 확인)
- [ ] cross-origin 요청이 CSRF 방어에 의해 차단됨
- [ ] 5회 로그인 실패 시 30분간 계정 잠김
- [ ] 30분 비활동 후 자동 로그아웃

### OWASP 기준 매핑
| 항목 | OWASP Cheat Sheet | 적용 여부 |
|------|-------------------|-----------|
| Access Token 저장 | Session Management | ✅ 메모리 저장 |
| Refresh Token 저장 | Session Management | ✅ HttpOnly Cookie |
| CSRF 방어 | CSRF Prevention | ✅ Token + 헤더 검증 |
| 세션 만료 | Session Management | ✅ 30분 비활동 |
| 로그인 잠금 | Authentication | ✅ 5회 실패 시 잠금 |

| P4-2 | Rate Limit 구현 | 등급별 Rate Limit 로직 | BE | 1일 | rate-limit.ts |
| P4-3 | Rate Limit 미들웨어 | API 엔드포인트별 적용 | BE | 0.5일 | 미들웨어 |
| P4-4 | 보안 헤더 | CSP, HSTS, X-Frame-Options 등 | BE/Infra | 0.5일 | 보안 헤더 설정 |
| P4-5 | CSP 정책 | Content-Security-Policy 설정 | FE | 0.5일 | CSP 설정 |

### 성공 기준
- [ ] Redis 기반 Rate Limit이 멀티 인스턴스에서 일관되게 동작
- [ ] 무료/Pro/Enterprise 등급별 다른 Rate Limit 적용
- [ ] 429 응답 시 Retry-After 헤더 포함
- [ ] 보안 헤더가 모든 응답에 포함됨 (securityheaders.com 점수 A)

### Rate Limit 정책
| 등급 | 분당 요청 | 적용 범위 |
|------|-----------|-----------|
| 무료 | 100 | 일반 API |
| Pro | 1,000 | 일반 API |
| Enterprise | 10,000 | 일반 API |
| 인증 | 5 | 로그인/회원가입 |
