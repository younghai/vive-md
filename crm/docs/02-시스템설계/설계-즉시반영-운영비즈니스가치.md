# 시스템 설계: 즉시 반영 + 운영 비즈니스 가치

## 1. 설계 목표
기존 API 구조를 바꾸지 않고, 현재 인증/권한 체계와 Soft Delete 패턴에 다음 기능을 덧붙인다.

- 권한 변경 즉시 반영
- 삭제 데이터 수동 복구 API
- 운영 헬스 체크 및 경량 지표

## 2. 아키텍처 영향

### 2.1 권한 즉시 반영
- 기존: Access Token payload의 `role` 사용
- 변경: Access Token은 신원 검증용으로 유지하고, 매 요청마다 `userId`로 DB의 최신 `role` 조회
- 효과: 관리자 권한 변경 즉시 반영, 재로그인 의존도 감소

### 2.2 관리자 권한 API
- 위치: `modules/auth`
- 신규 엔드포인트
  - `GET /api/v1/auth/admin/users` : 사용자 목록
  - `PATCH /api/v1/auth/admin/users/:id/role` : 역할 변경
- 게이트: `authMiddleware` + `adminOnly`

### 2.3 Soft Delete 복구
- 대상 모듈
  - `contacts`, `deals`, `tasks`
- 추가 엔드포인트
  - `POST /:id/restore` (각 모듈)
- 삭제 규칙
  - `deletedAt != null` 이어야 복구 허용
  - 소유자 또는 ADMIN만 허용

### 2.4 운영 헬스/메트릭
- 위치: `modules/health`
- 신규 라우트
  - `GET /api/v1/health/ready` : DB 접속 여부(실패 시 503)
  - `GET /api/v1/health/live` : 프로세스 생존성
  - `GET /api/v1/health/metrics` : 메모리/uptime/DB 상태

## 3. 보안/안정성 관점
- 인증 경로 예외(`register/login/refresh`) 및 헬스 경로는 기존 정책 유지.
- 권한 변경 경로는 `adminOnly`로 1차 막고, 객체 소유권은 개별 리소스 핸들러에서 2차 검증.
- `/health/ready`는 운영 감시에서 장애 전파를 빠르게 감지할 수 있도록 503 응답 제공.

## 4. 데이터 영향
- 스키마 변경 없음 (운영 부담 최소화)
- 기존 Soft Delete 정책과 호환되며 복구 시 `deletedAt`을 null로 리셋
