# 장애 대응 런북

> VIVE CRM 운영 환경에서 장애 발생 시 공통 대응 기준을 정의한다.

## 1. 목적과 범위

- 목적: 장애 인지부터 복구/사후 분석까지 표준 절차를 일관되게 적용
- 범위: API(`apps/api`), Web(`apps/web`), DB(PostgreSQL), 문서 계약(OpenAPI/Postman)

## 2. 장애 등급 정의

| 등급 | 기준 | 예시 | 목표 대응 |
|------|------|------|----------|
| SEV-1 | 전체 서비스 핵심 기능 중단 | 로그인 불가, API 5xx 급증 | 15분 내 완화 시작 |
| SEV-2 | 일부 핵심 기능 장애 | 딜 저장 실패, 리포트 API 오류 | 30분 내 완화 시작 |
| SEV-3 | 비핵심/우회 가능 장애 | 일부 UI 깨짐, 지표 오표시 | 영업일 내 조치 계획 |

## 3. 공통 대응 절차

1. 탐지: 모니터링/사용자 제보/헬스체크 알림 수신
2. 분류: 영향 범위와 등급(SEV) 판정
3. 완화: 트래픽 차단, 롤백, 기능 플래그 다운 등 즉시 조치
4. 복구: 정상 동작 확인 (health, 주요 API smoke test)
5. 사후: 원인 분석(RCA), 재발 방지 항목 등록

## 4. 시나리오별 즉시 조치

### 4.1 API 5xx 급증

1. `GET /healthz`, `GET /api/v1/health/ready`, `GET /api/v1/health/metrics` 확인
2. 최근 배포 여부 확인 후 필요 시 직전 릴리스로 롤백
3. DB 연결 실패 시 DB 상태/접속 정보(`DATABASE_URL`) 점검
4. 완화 후 핵심 엔드포인트 smoke test

### 4.2 인증 장애 (401/403 비정상 급증)

1. `JWT_SECRET`, 만료 정책(`JWT_ACCESS_EXPIRES_IN`) 변경 이력 확인
2. `auth` 라우트(`/api/v1/auth/login`, `/refresh`, `/me`) 동작 확인
3. 권한 관련 변경 직후라면 관리자 권한 변경 API 로그 점검

### 4.3 DB 장애 또는 성능 저하

1. `/api/v1/health/ready`가 503인지 확인
2. DB 커넥션/슬로우쿼리/락 상태 점검
3. 임시 완화: 고비용 리포트 호출 제한, 읽기 중심 기능 우선 복구

## 5. 복구 확인 체크리스트

- [ ] API ready/liveness 정상
- [ ] 로그인/토큰 갱신 정상
- [ ] Contacts/Deals/Tasks CRUD 정상
- [ ] 대시보드/리포트 기본 조회 정상
- [ ] OpenAPI 경로와 실제 라우트 계약 위반 없음
