## 1. 유지보수 유형 정의

### 1.1 유지보수 유형 분류

#### 수정 유지보수 (Corrective Maintenance)

| 항목 | 내용 |
|------|------|
| **정의** | 운영 중 발견된 결함(버그)을 수정하는 활동 |
| **목적** | 시스템의 정상 동작을 복구하고 사용자에게 미치는 영향을 최소화 |
| **트리거** | 사용자 보고, 모니터링 알림, 낮부 발견 |
| **우선순위** | 결함 심각도에 따라 즉시 ~ 차기 릴리스 |

**예시**:
- AI 리드 스코어링 결과가 표시되지 않는 버그 수정
- 고객 타임라인 데이터 누락 오류 수정
- 파이프라인 보드에서 딜 이동 시 데이터 정합성 오류 수정
- 브라우저 호환성 문제로 인한 UI 깨짐 수정
- 알림 발송 실패 오류 수정

#### 적응 유지보수 (Adaptive Maintenance)

| 항목 | 내용 |
|------|------|
| **정의** | 변화하는 환경(OS, 브라우저, 라이브러리, 규정 등)에 시스템을 적응시키는 활동 |
| **목적** | 변경된 환경에서도 시스템이 정상적으로 동작하도록 보장 |
| **트리거** | 외부 환경 변화 (OS 업데이트, API 변경, 규정 변경 등) |
| **우선순위** | 환경 변화 시점 및 영향도에 따라 결정 |

**예시**:
- OpenAI API 버전 업그레이드 대응
- Next.js/Vercel 플랫폼 업데이트 대응
- PostgreSQL 버전 업그레이드 대응
- 개인정볼보호법 변경에 따른 데이터 처리 방식 수정
- 브라우저 보안 정책 변경 대응 (쿠키 SameSite 등)

#### 완전 유지보수 (Perfective Maintenance)

| 항목 | 내용 |
|------|------|
| **정의** | 사용자 요구사항 변경이나 기능 개선을 통해 시스템을 향상시키는 활동 |
| **목적** | 사용자 만족도 향상 및 비즈니스 가치 증대 |
| **트리거** | 사용자 피드백, 비즈니스 요구, 경쟁 대응 |
| **우선순위** | 비즈니스 가치 및 사용자 영향도에 따라 결정 |

**예시**:
- AI 추천 알고리즘 정확도 개선
- 대시보드 성능 최적화 (쿼리 응답 시간 개선)
- UX 개선: 파이프라인 보드 인터랙션 향상
- 새로운 기능 추가: 이메일 통합, 웹폼 리드 캡처
- 모바일 반응형 UI 개선
- 접근성 개선 (WCAG 준수)

#### 예방 유지보수 (Preventive Maintenance)

| 항목 | 내용 |
|------|------|
| **정의** | 잠재적 문제를 사전에 식별하고 예방하기 위한 활동 |
| **목적** | 시스템 안정성 확보 및 장애 예방 |
| **트리거** | 정기 점검, 기술 부채 검토, 보안 감사 |
| **우선순위** | 위험도 및 영향도에 따라 결정 |

**예시**:
- 코드 리팩토링 (복잡도 감소, 가독성 향상)
- 의존성 패키지 정기 업데이트 (npm audit)
- 데이터베이스 인덱스 최적화
- 보안 취약점 사전 점검 및 패치
- 테스트 코드 보강
- 기술 부채 해소

### 1.2 유지보수 유형별 비율 목표

| 유지보수 유형 | 목표 비율 | 비고 |
|-------------|-----------|------|
| 수정 (Corrective) | 20% | 결함 수정에 집중 |
| 적응 (Adaptive) | 25% | 환경 변화 대응 |
| 완전 (Perfective) | 40% | 기능 개선에 가장 큰 투자 |
| 예방 (Preventive) | 15% | 안정성 유지 |

## 2. 변경 요청(CR) 프로세스

### 2.1 CR 프로세스 흐름도

```mermaid
flowchart TD
    A[변경 요청 접수] --> B[CR 등록 및 분류]
    B --> C[영향 분석]
    C --> D{영향도 / 규모 판단}

    D -->|소규모 변경| E1[팀 리드 승인]
    D -->|중규모 변경| E2[변경 관리 위원회 (CCB) 검토]
    D -->|대규모 변경| E3[경영진 승인]

    E1 --> F{승인?}
    E2 --> F
    E3 --> F

    F -->|승인| G[구현 계획 수립]
    F -->|반려| H[요청자 피드백 전달]
    F -->|보류| I[보류 사유 기록 및 재검토 일정 설정]

    H --> J[종료 또는 재요청]
    I --> B

    G --> K[구현]
    K --> L[테스트]
    L --> M{테스트 통과?}
    M -->|예| N[배포]
    M -->|아니오| K

    N --> O[검증 및 모니터링]
    O --> P[변경 완료 및 종료]
```

## 3. SLA 정의

### 3.1 장애 등급별 대응/복구 시간 목표

| 장애 등급 | 정의 | 대응 시간 | 우회 방안 제공 | 복구 시간 | 근본 원인 분석 |
|-----------|------|-----------|---------------|-----------|---------------|
| **Critical (P1)** | 전체 서비스 중단, 핵심 기능 불가, 데이터 유실 | 15분 | 1시간 | 4시간 | 3영업일 |
| **Major (P2)** | 주요 기능 장애, 심각한 성능 저하 | 30분 | 2시간 | 8시간 | 5영업일 |
| **Minor (P3)** | 부가 기능 장애, 경미한 성능 저하 | 2시간 | 4시간 | 3영업일 | 10영업일 |
| **Low (P4)** | 사용에 영향 없는 경미한 문제 | 1영업일 | 필요 시 | 10영업일 | 필요 시 |

> **대응 시간**: 장애 인지 후 담당자가 분석에 착수하기까지의 시간
> **복구 시간**: 장애 인지 후 서비스가 정상 운영 상태로 복구되기까지의 시간

## 4. 기술 부채 관리

### 4.1 기술 부채 분류

| 분류 | 설명 | 예시 | 영향 |
|------|------|------|------|
| **코드 부채** | 코드 품질 저하, 복잡도 증가, 중복 코드 | 하드코딩된 값, 매직 넘버, 긴 함수, 복잡한 조건문 | 유지보수 비용 증가, 버그 발생률 증가 |
| **아키텍처 부채** | 부적절한 설계, 확장성 제한 | Modular Monolith의 모듈 경계 모호, 부적절한 의존성 | 확장 어려움, 성능 병목 |
| **테스트 부채** | 테스트 부족, 테스트 품질 저하 | 낮은 코드 커버리지, E2E 테스트 부재, 깨진 테스트 방치 | 회귀 버그 위험, 리팩토링 어려움 |
| **문서 부채** | 문서 부재 또는 오래된 문서 | API 문서 미갱신, 운영 매뉴얼 부재, 주석 미갱신 | 온보딩 어려움, 운영 실수 |
| **인프라 부채** | 인프라 설정 관리 부재, 수동 운영 | IaC 미적용, 수동 배포, 모니터링 부족 | 운영 실수, 복구 지연, 확장 어려움 |
