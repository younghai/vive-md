## 3. 배포 프로세스

### 3.1 배포 정책

| 환경 | 브랜치 | 트리거 | 승인 필요 |
|------|--------|--------|-----------|
| Staging | `develop` | PR Merge | 없음 |
| Production | `main` | Manual | 1명 이상 |

### 3.2 배포 체크리스트

#### Frontend (Next.js / Vercel)

```bash
# 1. 환경 변수 확인
vercel env ls

# 2. 빌드 검증
npm run build

# 3. 린트 검사
npm run lint

# 4. 타입 검사
npm run type-check

# 5. 배포
vercel --prod
```

#### Backend (Hono / Railway/Fly.io)

```bash
# 1. 환경 변수 확인
railway variables

# 2. 마이그레이션 검증
npm run db:generate
npm run db:migrate:check

# 3. 빌드 검증
npm run build

# 4. 배포
railway up
```

### 3.3 데이터베이스 마이그레이션

```bash
# 1. 스키마 변경 확인
npm run db:diff

# 2. 마이그레이션 파일 생성
npm run db:generate

# 3. Staging 적용
npm run db:migrate:staging

# 4. Production 적용 (수동)
npm run db:migrate:prod
```

## 5. 장애 대응

### 5.1 장애 등급 정의

| 등급 | 정의 | 예상 영향 | 응답 시간 |
|------|------|-----------|-----------|
| **P0 (심각)** | 서비스 중단 | 모든 사용자 영향 | 15분 이내 |
| **P1 (고)** | 핵심 기능 장애 | 주요 기능 사용 불가 | 30분 이내 |
| **P2 (중)** | 일부 기능 장애 | 특정 기능 사용 불가 | 2시간 이내 |
| **P3 (저)** | 경미한 문제 | UX 저하 | 다음 영업일 |

### 5.2 장애 시나리오별 대응

#### 시나리오 1: 서버 다운 (애플리케이션 응답 없음)

```
[증상]
- API 응답 없음 (Connection refused/timeout)
- Health check 실패
- Error rate 100%

[진단 절차]
1. Railway/Fly.io 대시보드 확인
2. 배포 로그 확인 (최근 배포 여부)
3. 컨테이너 상태 확인
4. 리소스 사용량 확인

[복구 절차]
1. 컨테이너 재시작
   railway restart
   
2. 롤백 (최근 배포가 원인일 경우)
   railway rollback
   
3. 스케일업 (리소스 부족일 경우)
   railway scale --service api --replicas 2

[사후 조치]
- 장애 원인 분석
- 모니터링 개선
- 문서 업데이트
```

#### 시나리오 2: DB 커넥션 풀 소진

```
[증상]
- API 응답 지연 (타임아웃)
- "too many connections" 에러
- DB CPU/메모리 정상

[진단 절차]
1. 현재 연결 수 확인
   SELECT count(*) FROM pg_stat_activity;
   
2. 유휴 연결 확인
   SELECT * FROM pg_stat_activity WHERE state = 'idle';
   
3. 장기 실행 쿼리 확인
   SELECT * FROM pg_stat_activity WHERE state = 'active' 
   AND now() - query_start > interval '30 seconds';

[복구 절차]
1. 유휴 연결 종료
   SELECT pg_terminate_backend(pid) FROM pg_stat_activity 
   WHERE state = 'idle' AND state_change < now() - interval '10 minutes';
   
2. 커넥션 풀 크기 조정 (임시)
   DATABASE_URL에 connection_limit 파라미터 조정
   
3. 애플리케이션 재시작

[사후 조치]
- 커넥션 풀 설정 최적화
- 쿼리 성능 개선
- 연결 누수 검사
```

#### 시나리오 3: 디스크 풀

```
[증상]
- "No space left on device" 에러
- 로그 쓰기 실패
- 임시 파일 생성 불가

[진단 절차]
1. 디스크 사용량 확인
   df -h
   
2. 큰 파일/디렉토리 확인
   du -sh /* | sort -hr | head -20
   
3. 로그 파일 확인
   ls -lh /var/log/

[복구 절차]
1. 로그 로테이션
   sudo logrotate -f /etc/logrotate.conf
   
2. 오래된 로그 삭제
   find /var/log -name "*.log.*" -mtime +7 -delete
   
3. 임시 파일 정리
   sudo rm -rf /tmp/*
   
4. (DB) WAL 아카이브 정리
   SELECT pg_archive_cleanup('...', '...');

[사후 조치]
- 디스크 용량 확장
- 로그 보관 정책 검토
- 모니터링 알림 설정
```

#### 시나리오 4: AI API 장애 (외부 연동)

```
[증상]
- AI 기능 응답 없음/타임아웃
- 외부 API 5xx 에러
- Rate limit 초과

[진단 절차]
1. 외부 API 상태 페이지 확인
   - OpenAI Status: https://status.openai.com
   - Claude Status: https://status.anthropic.com
   
2. API 응답 시간 확인
   curl -w "@curl-format.txt" -o /dev/null -s [API_ENDPOINT]
   
3. Rate limit 확인
   응답 헤더: x-ratelimit-remaining

[복구 절차]
1. Fallback 모드 활성화
   - 캐시된 응답 사용
   - 간소화된 응답 제공
   
2. Circuit Breaker 동작 확인
   - 자동 failover 확인
   
3. 대체 API 전환 (설정된 경우)
   - Secondary provider로 전환
   
4. Retry 로직 확인
   - Exponential backoff 동작 확인

[사후 조치]
- 외부 API SLA 검토
- Fallback 로직 개선
- Circuit breaker 설정 조정
```

## 6. 백업 및 복구

### 6.1 백업 정책

| 대상 | 방식 | 주기 | 보관 기간 | 저장 위치 |
|------|------|------|-----------|-----------|
| PostgreSQL | 전체 백업 | 일 1회 (03:00) | 30일 | S3 (R2) |
| PostgreSQL | WAL 아카이빙 | 지속 | 7일 | S3 (R2) |
| Redis | 스냅샷 | 일 1회 | 7일 | Upstash 자동 |
| 파일 첨부 | 증분 백업 | 실시간 | 90일 | R2 |

### 6.2 RTO / RPO

| 항목 | 목표 값 | 설명 |
|------|---------|------|
| **RTO** | 1시간 | Recovery Time Objective - 복구 소요 시간 |
| **RPO** | 5분 | Recovery Point Objective - 데이터 손실 허용 범위 |

### 6.3 복구 절차

#### PostgreSQL 복구

```bash
# 1. 백업 파일 목록 확인
aws s3 ls s3://vive-crm-backups/postgresql/ --recursive | tail -10

# 2. 최신 백업 다운로드
aws s3 cp s3://vive-crm-backups/postgresql/backup_YYYYMMDD_HHMMSS.sql.gz ./

# 3. 복구용 DB 생성
createdb vive_crm_restore

# 4. 복구 실행
gunzip < backup_YYYYMMDD_HHMMSS.sql.gz | psql vive_crm_restore

# 5. (PITR 필요시) WAL 적용
# pg_basebackup + WAL 아카이브 조합

# 6. 애플리케이션 설정 업데이트
# DATABASE_URL 변경

# 7. 연결 확인
psql $DATABASE_URL -c "SELECT version();"
```

#### Redis 복구

```bash
# Upstash는 자동 복구 지원
# 수동 복구 필요시:

# 1. 최신 RDB 파일 확인 (Upstash Console)

# 2. 새 인스턴스 생성 또는 기존 인스턴스 복원

# 3. 애플리케이션 재시작하여 새 엔드포인트 연결
```

### 6.4 백업 검증

```bash
# 주간 백업 검증 (자동화 권장)

# 1. 테스트용 복구 환경 준비
# 2. 백업 파일 무결성 검사
gunzip -t backup_YYYYMMDD_HHMMSS.sql.gz

# 3. 테스트 복구 수행
# 4. 일부 데이터 쿼리 검증
# 5. 결과 보고
```
