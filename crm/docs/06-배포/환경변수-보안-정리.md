# 환경변수 · 보안 설정 정리

이 문서는 CRM API 구동에 필요한 환경변수 구조를 정리한 운영 기준입니다.

## 1. 파일 역할 정리

- `apps/api/.env.example`
  - API 런타임용 기본 템플릿
  - JWT 비밀값, DB URL, 허용 Origin을 선언하는 기준 샘플
  - **반드시 버전 관리에 남김**

- `apps/api/.env`
  - 로컬 개발용 실제 값(로컬 기본값 또는 override)
  - Git에서 추적 대상에서 제외되어야 함

- `.env.local.example`
  - 루트 레벨 공용 로컬 기준 예시 (문서/운영 참고)

- `.env.production.example`
  - 루트 레벨 운영 기준 예시 (문서/운영 참고)

- `.env.production`, `.env.local`
  - 실제 운영/로컬 주입 값으로만 사용하거나 임시 테스트 용도로 사용
  - 실 운영 비밀은 플랫폼 Secret/비밀관리서비스로 분리 권장

## 2. 필수 보안 변수

- `JWT_SECRET`
  - 32자 이상 권장
  - JWT 서명/검증에 직접 사용되므로 유출 시 즉시 교체 필요
- `DATABASE_URL`
  - `postgresql://user:password@host:5432/dbname?schema=public`
  - 비밀번호는 평문 보관 금지
- `CORS_ORIGINS`
  - 환경별 프론트 도메인만 허용 (개발: localhost, 운영: 실제 도메인)

## 3. 권장 운영 규칙

1. `.env`, `.env.local`, `.env.production`, `apps/api/.env` 등 실제 값 파일은 Git에 커밋하지 않는다.
2. 예시 파일은 `.env.example`, `.env.local.example`, `.env.production.example`만 공유한다.
3. 운영 비밀값(`JWT_SECRET`, DB 비밀번호, 외부 API 키)은
   - Vercel/Railway Secret Manager
   - GitHub Actions Secret
   - Vault 같은 시크릿 저장소
   를 통해 주입한다.
4. 배포 전 필수 변수 점검:
   - `NODE_ENV`
   - `DATABASE_URL`
   - `JWT_SECRET`
   - `JWT_ACCESS_EXPIRES_IN`
   - `JWT_REFRESH_EXPIRES_DAYS`
   - `CORS_ORIGINS`

### 실행 체크

```bash
cd apps/api
npm run env:check
```

## 4. 비밀값 생성 예시

```bash
# 32바이트 랜덤 키
openssl rand -base64 32

# 또는 Node.js
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
```

## 5. 예시 파일 생성 규칙

- 새 환경 추가 시:
  1. `.env.example`을 복제해 `.env` 또는 앱별 환경 파일 생성
  2. 필요 시 운영 플랫폼 환경변수로 덮어쓰기
  3. 민감값은 주입 후 즉시 `env` 파일 삭제 또는 보관 경로 분리

## 6. docker-compose 연동

- `apps/api/docker-compose.yml`은 런타임 값 주입 시 `apps/api/.env`를 기본으로 사용하고,
  필요 시 DB 호스트 등 런타임 차이를 환경변수로 오버라이드한다.
